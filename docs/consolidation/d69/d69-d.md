# D69-d — Surface is primary; agent adapts to the surface

> Draft option D for D69. The most divergent option; deliberately tests
> whether D68's "same workflow" claim can hold. One of multiple options
> being explored before promoting one to DECISIONS.md.

## Caveat — this option contradicts D68 by construction

D68 (firm) declares "multi-surface product, one fixed workflow." This
option reverses that: each surface is its own product with its own
workflow, and "Agentera" is a brand family rather than a single
product.

The rest of this draft is written assuming D68 is revisable. If D68
holds, this option is rejected by contradiction — see
[Confidence](#confidence) for the explicit position.

## Question

How should Agentera's surfaces (cli, mobile, web, desktop, tui, editor
skill/plugin runtimes) be built around the same core capabilities and
workflow, so the experience is consistent and familiar but adapted to
each surface?

## Context

- **D67** (2026-06-05, firm) — single brand everywhere; mobile is the
  flagship surface.
- **D68** (2026-06-05, firm) — multi-surface product, one fixed
  workflow. No user extensions.

## Design anchors

1. **The surface is the product; the agent is a feature of the
   product.** The product lives on the surface. The capability engine
   is a library the product uses, not the product itself. A user buys
   a surface, not "Agentera" in the abstract.
2. **Brand is the only invariant across surfaces.** Shared across
   surfaces: the name, the visual identity (glyph, color, typography
   from `protocol.yaml`), the `.agentera/` directory convention, and
   the publishing namespace (`@agentera/cli`, `@agentera/mobile`,
   `@agentera/web`). Not shared: workflow, capability count,
   capability behavior, state shape, command names, product
   positioning.
3. **Capability roster: per-surface, by design.** A surface ships the
   capabilities that fit its shape. CLI ships the full twelve; mobile
   ships a touch-fit subset; web ships a docs-and-dashboard subset.
   Overlap is incidental, not required.
4. **Capability schema: surface-defined.** Two surfaces may ship a
   capability of the same name with different behavior. CLI realisera
   is a full autonomous execution loop. Mobile realisera is a guided
   assistant that hands the plan off to another surface. They are
   linked by name and brand, not by a shared behavior contract.
5. **Continuity: brand-level only.** A user using Agentera on mobile
   and Agentera on CLI is using two products that share a name. No
   guarantee that the same capability name does the same thing.
   Cross-surface handoff is a deliberate user action, not an ambient
   property.

## Implication

The surface is the product. Agentera is a family of products under
one brand, much the way Adobe ships Photoshop, Illustrator, and
Acrobat under one roof without pretending they share a workflow.

### What this looks like, by surface

- **Mobile (`@agentera/mobile`, SvelteKit, touch-first).** A
  thinking partner for talking through work, not for executing it.
  Roster: `hej`, `planera`, `realisera` (guided — produces a plan the
  user carries to another surface to execute), `inspirera`. No
  terminal execution, no autonomous loops, no skill loading, no MCP,
  no plugin extensions. Mobile is opinionated on purpose; the
  "no peer" anchor means it does not need a CLI under the hood to
  ship value.
- **CLI (`@agentera/cli`, terminal-first).** The full engineering
  toolkit. All twelve capabilities in autonomous mode. Realisera
  runs in an event loop against the filesystem. This is also the
  canonical runtime that editor plugins and any thin client route
  through — the editor's "agent" is a view onto this CLI. Single
  binary Node distribution; `npx -y agentera@latest` / `@next`
  channels.
- **Web (`@agentera/web`, Astro + Starlight, Cloudflare).**
  Documentation site plus a lightweight dashboard. Roster: `hej`,
  `visionera`, `planera`, `realisera` (review-only — a read view of
  what CLI realisera produced), `profilera`, `dokumentera`.
  Static-first; no agent loop in the browser.
- **Editor plugin (Cursor, VS Code).** A sidebar panel that defers
  to the CLI runtime. The plugin is a thin view: it shows
  `.agentera/` state, surfaces the CLI's commands, renders output.
  It does not own an autonomous loop. Realisera in the editor is a
  view onto CLI realisera, not a sibling.

### What stays shared, concretely

- Visual identity (glyph, color, typography).
- The `.agentera/` directory name and rough on-disk shape (the CLI
  is the canonical writer; other surfaces read it).
- The publishing brand and package namespace.
- The capability **naming** vocabulary (`realisera`, `planera`, …)
  — not the capability **behavior**.

## Trade-offs

- (+) **Most surface freedom.** Each surface is a genuine, full
  product. No surface is a shell; no surface is constrained by
  another's design. A surface team can ship a feature that only
  makes sense on that surface without negotiation.
- (+) **Maximum surface-native UX.** No translation cost. Mobile
  UX is designed for mobile; CLI UX is designed for the terminal.
  No lowest-common-denominator interaction model.
- (+) **Easier to ship per surface.** A new surface does not need
  a global workflow change. Add it with its own workflow; the rest
  of the family is unaffected.
- (+) **Honest about reality.** If two surfaces are doing genuinely
  different things, calling them one product with "one fixed
  workflow" is a fiction the user can feel.
- (−) **Contradicts D68 directly.** "One fixed workflow" is the
  bedrock of D68; this option reverses it. Adopting D requires
  revisiting D68 or formally superseding it. This is the headline
  cost, not a footnote.
- (−) **The "consistent and familiar" promise thins out.** What
  stays familiar is the brand and the directory name, nothing
  internal. A user who learns realisera on the CLI is not
  guaranteed anything about realisera on mobile.
- (−) **Cross-surface continuity is unreliable.** Users cannot
  assume "realisera does X here because it did X there." Each
  surface must be learned.
- (−) **Discovery and documentation cost.** A user has to learn
  what each surface does differently; the docs cannot point to
  "realisera" as a single global concept. A surface matrix
  replaces a single capability reference page.
- (−) **Brand-dilution risk.** If the surfaces diverge enough, the
  brand becomes a marketing shell with no operational meaning,
  and the suite fragments into independent products that happen
  to share a logo.

## Open questions

- **What does "same Agentera" mean if the workflows diverge?**
  *Brand, visual identity, on-disk convention, and the capability
  naming vocabulary. Nothing else. The "product" lives on each
  surface; "Agentera" is the publisher, not the product.*
- **How do you prevent capability naming from drifting across
  surfaces (e.g., CLI realisera ≠ mobile realisera)?** *Shared
  name vocabulary at the schema level (`realisera` always ships
  under that name) but each surface's `instructions.ts` is free
  to define what it means. Governance, not enforcement. A
  real cost; the surface team owns the answer.*
- **Is this still a "multi-surface product" per D68, or is it a
  product family that needs D68 to be revisited?** *This option
  is the latter. Adopting D requires revising D68, not working
  around it. The contradiction is honest and explicit.*
- **How does the user discover what each surface does
  differently?** *Per-surface docs and a one-page surface
  matrix. No global capability reference; the capability is a
  per-surface contract.*
- **What is the "agent" in "Agentera" if each surface is its
  own product?** *A brand-level concept, not a runtime. The
  runtime is the CLI; the brand is "we ship thinking-tool
  products that share a name."*

## Confidence

`provisional` — explicitly a live candidate, but its acceptance
requires revising D68. The option serves three purposes at once:

1. **A serious proposal** for a brand-family architecture if D68
   is in fact the wrong call.
2. **A pressure test** for D69-a, D69-b, and D69-c. If those
   three options hold up under the same scrutiny, D68's "one
   fixed workflow" claim is on solid ground; if they don't, D68
   may be the one that needs revising.
3. **A cost benchmark.** Whatever D69 picks, the suite should be
   able to point at D and say "we considered this and the cost
   is X." That requires D to be a real option on the page, not
   a strawman.

If D68 stays firm after D69 lands, D is rejected by contradiction
without further debate. If D68 is reopened, D is the most
divergent live candidate and earns the most rigorous test.
