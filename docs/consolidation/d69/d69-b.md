# D69-b — User journey is the load-bearing level

> Draft option B for D69. One of multiple options being explored before
> promoting one to DECISIONS.md.

## Question

How should Agentera's surfaces (cli, mobile, web, desktop, tui, editor
skill/plugin runtimes) be built around the same core capabilities and
workflow, so the experience is consistent and familiar but adapted to each
surface?

## Context

- **D67** (2026-06-05, firm) — single brand everywhere; mobile is the
  flagship surface.
- **D68** (2026-06-05, firm) — multi-surface product, one fixed workflow.
  No user extensions.

## Design anchors

1. **User journey is the invariant.** The 5-phase model (orient → plan →
   build → verify → close) is the level at which every surface
   recognizes itself as "doing Agentera." It is an abstraction over the
   12 named capabilities (one working mapping: orient ≈ visionera + hej;
   plan ≈ planera; build ≈ realisera (execution); verify ≈ inspektera;
   close ≈ realisera (release) + dokumentera). Capability **schemas**
   and the state contract **are not** the invariant; they are
   surface-shaped. A surface is allowed to compress or merge phases
   (e.g., mobile may collapse orient + plan into a single "what now"
   step), but it may not drop the sequence, rename the phases, or
   introduce a phase that breaks the closing arc.
2. **Capability set is a surface-defined subset.** Each surface ships
   the capabilities that fit its affordances. Mobile may ship a slim
   realisera; CLI ships the full set; web may omit orkestrera. The
   subset is a deliberate product decision, not a constraint imposed
   by a shared runtime.
3. **State contract is a per-surface YAML subset.** Each surface owns
   the `.agentera/` shape it needs. CLI may carry a full `vision.yaml`,
   `plan.yaml`, `progress.yaml`; mobile may keep a SQLite-backed
   compact slice that round-trips losslessly with the canonical YAML
   on handoff.
4. **Surface autonomy is over expression, not substance.** Each
   surface is a first-class product that shapes the **affordances**
   through which the user experiences the journey (touch vs keyboard,
   inline vs modal, sync vs async, verbose vs terse). It does **not**
   reshape the journey itself. A surface that wants to drop the
   "close" phase or invent a "live pair" phase is out of scope for
   D69-b; that is a new journey, not a new surface of this one.
5. **Continuity is the journey's recognizability, not state sync.**
   The user moving from mobile to CLI is "still in the build phase,"
   not necessarily "loading the same in-flight task." State handoff
   between surfaces is allowed and encouraged where feasible, but it
   is a separate concern from journey continuity. The journey is the
   part that survives the handoff; the state is the part that may
   need to be re-fetched, re-derived, or simply re-entered.

## Implication

The user journey is the product. Surfaces are first-class products that
share a mental model.

- Mobile's realisera is a slimmed, touch-friendly version of CLI's
  realisera. Both serve the same mental phase, but neither is "wrong."
- `.agentera/` shapes differ; mobile might use SQLite-backed compact
  state, CLI uses the full YAML suite.
- The user is "doing the orient phase" everywhere — but the work itself
  adapts to the surface.
- The capability contract shifts: a "capability" is identified by the
  phase it serves, not by an exact behavioral-parity check. Two
  surfaces' realiseras are "the same capability" because they both
  serve the build phase, not because they produce identical output.

## Surface sketches

- **Mobile.** Affordance-driven, low-bandwidth. Likely ships orient
  (hej dashboard, visionera lite) + plan (planera) + close
  (dokumentera, release summary). Build and verify are read-only
  views into work the CLI or an editor is doing; the user monitors
  rather than drives them. State is a compact SQLite slice; full
  YAML is fetched on demand. Touch gestures map directly to phase
  transitions (swipe to advance, long-press to inspect).
- **CLI.** Full capability surface. All 12 capabilities reachable;
  full YAML state; scriptable. Keyboard-first. The "canonical"
  surface against which other surfaces define their subset and
  against which `agentera check validate` runs end-to-end.
- **Web (docs + marketing).** Lightweight presence. Orients the user
  toward the brand and the install path; may expose hej for read-only
  status if the user is authenticated. Not expected to ship build,
  verify, or close.
- **Editor plugin (e.g., Cursor).** Hybrid: heavy on build and verify
  (inline completion, diff review, run-tests-in-place) and light on
  plan and close. Speaks the same journey vocabulary but expresses
  it through editor primitives. State lives where the editor lives;
  `.agentera/` YAML is the export format, not the working store.

## Trade-offs

- (+) Each surface is a genuine product with its own affordances, not
  a thin I/O adapter around a single canonical agent.
- (+) Surface-native UX is allowed to be different in **substance**,
  not just input shape. The mobile experience is allowed to feel
  mobile-native, not "a CLI with big buttons."
- (-) **Central cost: there is no single realisera.** Two surfaces
  that ship a build-phase capability may behave differently in
  observable ways (different defaults, different verbosity, different
  review cadence). D69-b is explicitly trading capability parity for
  surface autonomy. This must be accepted, not wished away.
- (-) Cross-surface continuity is weaker at the state level. The user
  may lose in-flight work when switching surfaces, or the work may
  exist in surface-local state that does not round-trip cleanly.
  Continuity is preserved only at the level of "which phase am I in"
  and "what vocabulary am I using."
- (-) **"Same Agentera" becomes a brand and documentation problem,
  not a runtime guarantee.** Surfaces must use the same vocabulary,
  same icons, same phase names, and same onboarding arc to maintain
  the user's sense of one product. Drift is a real risk; it must be
  caught in design review, not in the runtime.
- (-) The 5-phase model is a simplification. Some workflows (pure
  research via inspirera; orchestration via orkestrera; live
  pair-programming) do not fit cleanly. The design must acknowledge
  that journey-fit is a judgment call, not a checklist, and that
  edge-case surfaces may need to be deferred or treated as
  out-of-scope for the first cut.

## What would falsify this design

- A surface whose user mental model cannot be expressed in 5 phases
  (e.g., a "live pair" mode with no discrete close). D69-b would
  have to either extend the journey or refuse the surface.
- A user-research signal that the same vocabulary across surfaces
  reads as "different products" rather than "same product, different
  shape." The brand and documentation burden is too high.
- Two surfaces' realiseras diverge enough that users complain about
  parity. The design's central cost becomes a recurring support cost.

## Open questions

- Which phases are the **minimum** a surface must cover to count as
  "shipping Agentera"? Candidate answer: orient + build, since those
  are the most user-visible. Confirm with a mobile spike.
- How is the user journey versioned, and what is the migration story
  when a phase is added, renamed, or split? Should the journey have
  its own `journey_version` field, or live under `protocol_version`?
  Renaming a phase is a breaking change for every surface.
- How do we test "same mental phase" across surfaces? Proposed
  mechanism: cross-surface walkthrough scripts that exercise each
  phase from two surfaces and compare vocabulary, icons, and
  affordances — not output.
- Does D69-b require a shared UX vocabulary spec (icons, phase
  names, onboarding copy, error wording) to keep the "same product"
  promise? If so, where does it live — `protocol.yaml`, a new
  `journey.yaml`, or the `web/` docs site?
- How do the 5 phases interact with capabilities that span phases
  (e.g., hej as the always-available dashboard; dokumentera as a
  close-phase helper that also shows up mid-build)? Are those
  cross-phase capabilities a separate tier in the model?

## Confidence

provisional — the design is internally consistent and the trade-offs
above are concrete, not paper risks. The central question (whether
surface autonomy is worth the loss of capability parity) is the right
question for D69 to decide, and d69-a is the natural contrast.
Promote to firm only after the mobile and editor-plugin surface
sketches above are validated with at least one working spike each,
and after the shared-vocabulary question is resolved.
