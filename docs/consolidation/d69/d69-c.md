# D69-c — State contract is the load-bearing level

> Draft option C for D69. One of multiple options being explored before
> promoting one to DECISIONS.md.

## Question

How should Agentera's surfaces (cli, mobile, web, desktop, tui, editor
skill/plugin runtimes) be built around the same core capabilities and
workflow, so the experience is consistent and familiar but adapted to each
surface?

## Context

- **D67** (2026-06-05, firm) — single brand everywhere; mobile is the
  flagship surface.
- **D68** (2026-06-05, firm) — multi-surface product, one fixed workflow.
  No user extensions.

## Design anchors

1. **State contract is the invariant.** The `.agentera/` schema — artifact
   shapes, `source_contract`, and validation rules — is the single narrow
   waist. Capability schema and user journey can differ per surface; the
   shape and meaning of state cannot.
2. **Capability set: per-surface subset.** Each surface ships the
   capabilities it can run. CLI ships all 12; mobile ships a curated
   subset; web ships read-only inspectors and links back to a write
   surface.
3. **Capability schema: surface-specific.** The realisera trigger word
   maps to a *surface-bound* capability, not a single global capability.
   CLI's realisera is the full autonomous development engine; mobile's is
   a guided step-through assistant; web has no realisera. Same word,
   surface-resolved behavior.
4. **Surface role: first-class product.** A surface is a product in its
   own right, not a thin client. It picks its own subset, owns its own
   UX, and ships under the same brand.
5. **Continuity: through state, not behavior.** `.agentera/` syncs
   across surfaces (local-first with sync). A decision made on mobile
   shows up on CLI; a CLI-authored progress update shows up on the
   mobile dashboard. Continuity is the state contract, not identical
   behavior.

## What this looks like, by surface

- **CLI** (all 12 capabilities, full autonomous). Power surface.
  Realisera runs end-to-end loops. Writes `.agentera/` directly. Local
  filesystem, terminal-native. Source of truth for behavioral capability
  definitions.
- **Mobile** (curated subset, guided). Realisera is a conversational
  guide that asks, proposes, and writes `decisions.yaml` and
  `progress.yaml`. Dokumentera is read-only with light edits. Mobile
  syncs `.agentera/` from a write surface; lightweight decision capture
  is the main on-device write.
- **Web** (read-mostly, link-out). Starlight docs, marketing, and a
  state inspector ("where is this project?"). Realisera and other write
  capabilities are not present; the UI surfaces a "Continue in CLI" or
  "Open in mobile" affordance wherever a write is implied.
- **Editor plugin (Cursor)** (curated subset, IDE-native). Realisera is
  triggered from the editor sidebar with access to files, terminal, and
  code-aware diff review. Behavior is closer to CLI's full autonomous
  mode but bounded by editor affordances (in-file review, no long
  detached processes).

## Implication

The data is the product. Surfaces are first-class products that share a
state contract.

- Mobile's realisera and CLI's realisera are *different capabilities
  bound to the same trigger word*, not the same capability with two
  implementations. The state they write to is identical and
  cross-validatable.
- `.agentera/` syncs across surfaces; one source of truth per user.
- The product *accepts* behavioral divergence across surfaces as a
  conscious trade for surface-native UX.

## Trade-offs

- (+) **Surface freedom.** Each surface picks its own UX, its own
  capability subset, and the affordances that fit its form factor.
- (+) **State as the continuity substrate.** Users move between surfaces
  without losing work; the contract guarantees meaning.
- (-) **"Realisera does X here but Y there" is real, not paper.** A user
  who runs realisera on CLI and then on mobile will see different
  behavior. The design accepts this cost; it is the central tension
  with the "single brand, one fixed workflow" stance from D67/D68.
- (-) **Drift risk is structural.** Without a binding mechanism, a
  surface's "realisera" can drift semantically from another's. A
  candidate mitigation is a per-surface capability manifest (e.g., a
  `surface.yaml` shipped with each surface) that declares what each
  trigger word means there; the design has not yet committed to this.

## Open questions

- **Capability surface manifest.** What does the per-surface declaration
  of "realisera v1 on this surface does X" look like, and where is it
  shipped (skill bundle? surface profile? registry entry)? This is the
  concrete mitigation for behavioral drift and the answer to the
  "how do you know both runs are 'realisera'?" version-compatibility
  question.
- **State schema versioning.** When `.agentera/` gains a new field, how
  does an older surface handle it — strict rejection, graceful
  degradation, or forward-fill defaults? This must be specified in the
  state contract itself, not left per-surface.
- **Surface onboarding.** How does a user learn what each surface's
  capabilities do? Candidate options: a `surface.yaml` consumed by the
  docs site to auto-render a per-surface capability map, or a first-run
  primer. The web surface is the natural home for this.

## Confidence

provisional — deliberate contrast to d69-a and d69-b; trade-offs are
real and the design has not yet been validated against the "single
brand" and "one fixed workflow" constraints from D67/D68.
