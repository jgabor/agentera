# D69-a — Capability schema is the load-bearing level

> Draft option A for D69. One of multiple options being explored before
> promoting one to DECISIONS.md.

## Question

How should Agentera's surfaces (cli, mobile, web, desktop, tui, editor
skill/plugin runtimes) be built around the same core capabilities and
workflow, so the experience is consistent and familiar but adapted to each
surface?

## Context

- **D67** (2026-06-05, firm) — single brand everywhere; mobile is the
  flagship surface.
- **D68** (2026-06-05, firm) — multi-surface product, one fixed workflow.
  Surfaces answer "where do I use Agentera?" not "what can I bolt on?"
  No user extensions.

## Design anchors

1. **Same workflow level: capability schema.** Capability behavior is
   identical across surfaces. UX adapts, agent behavior does not. A
   realisera run triggered from any surface produces the same artifact
   graph and decision log; surface-specific rendering is confined to
   input capture and output display.

2. **Trigger location: in the capability.** Each capability owns its
   canonical trigger name and argument shape in the schema. Surfaces
   resolve triggers to UI affordances (commands, buttons, slash
   commands) and pass the parsed argument set back to the runtime. A
   surface that invents a new trigger is a bug, not an extension.

3. **Surface role: thin I/O adapter.** A surface translates input,
   renders output, and locally caches state. It does not define
   capabilities, does not encode business rules, and does not own the
   canonical state. Any business logic that lives only on one surface
   is a defect.

4. **State location: local-first with sync.** Each surface reads and
   writes its own local `.agentera/`; sync is asynchronous and
   surfaces converge per a defined conflict-resolution policy (see
   Open questions). Mobile can run fully offline.

5. **In-flight continuity: same task across surfaces.** The in-flight
   task is part of project state, not surface-local session state. A
   realisera run started on mobile can be observed or resumed on CLI
   without losing context. Execution itself is hosted on a single
   owner per run (today: the Cloudflare Worker that mobile and web
   rely on); CLI is a local execution owner when it initiates a run.

## Implication

The agent is the product. Surfaces are shells around the same workflow.

- Mobile UX is touch-native; CLI UX is terminal-native. Both invoke
  the same realisera and observe the same decision log.
- `.agentera/` syncs across all surfaces; offline writes merge on
  next online tick.
- New capabilities are added to the schema once; all surfaces pick
  them up by reading the same contract.

## Trade-offs

These costs are accepted in exchange for the consistency D68 demands.
Each is a real liability that the anchors above do not pay for on
their own, and each should be visible to anyone implementing the
design.

- **Schema is a single point of failure.** A breaking change to the
  capability schema forces a coordinated update across every
  surface. Schema evolution must be versioned and migration-tested,
  not just appended to.
- **Local-first sync is a hard problem.** The conflict-resolution
  policy is the most likely place this design will need to grow.
  Decision log and plan likely need append-only semantics; free-form
  prose (`DOCS.md`, scratch notes) likely need a real merge
  function. The policy is open and is the design's largest
  unresolved cost.
- **In-flight handoff is bounded by the execution owner.** A run
  hosted on mobile's Cloudflare Worker can be observed and steered
  from CLI, but the actual computation does not migrate between
  machines mid-step. The design promises continuity of task state,
  not continuity of process state.
- **Thin adapters limit surface-native flourishes.** Combined with
  D68's "no extensions" rule, a UX win that only makes sense on
  one surface cannot ship there alone. Every surface-specific
  affordance must trace back to a capability in the schema.
- **Read-only surfaces still pay the schema tax.** Web (docs) does
  not write state, but it still consumes the schema to render
  capability descriptions. The schema is a publishing contract
  as well as a behavior contract; the cost is shared with surfaces
  that don't benefit from the local-first model.

## Concrete by surface

- **CLI** (`@agentera/cli`): the canonical surface and the reference
  implementation of "thin adapter." `agentera <capability>` is the
  trigger; the CLI owns the schema loader today and is what every
  other surface should match in adapter weight.
- **Mobile** (`@agentera/mobile`): capability triggers render as
  sidebar actions and slash commands in the SvelteKit UI. The app
  is offline-first against a local `.agentera/`; execution is
  hosted on a Cloudflare Worker that mobile drives. Handoff to CLI
  is shared task state, not process migration.
- **Web** (`@agentera/web`): primarily a publishing surface.
  Starlight docs are generated from the capability schema, so a
  new capability updates docs without a separate authoring step.
  The web app is not a state-writing surface in this design.
- **Editor plugin** (Cursor, VS Code): consumes the schema and
  surfaces triggers as commands and keybindings. The editor's
  workspace `.agentera/` is shared with the CLI, which makes the
  editor the most natural home for "start here, finish there"
  handoffs because the workspace is already a single identity.
- **Desktop and TUI**: not yet on the roadmap but, in this
  design, would be additional thin adapters — desktop as a
  native shell around the CLI and schema, TUI as a terminal
  mode of the same adapter. Neither gains new capability
  authority.

## Open questions

- Conflict resolution policy when state syncs — per-artifact
  last-writer-wins vs. CRDT vs. schema-defined merge function.
  The right answer probably differs by artifact shape, and
  "DOCS.md edits on mobile and CLI in parallel" is the
  motivating failure case.
- Whether the schema's current home
  (`capability_schema_contract.yaml` plus
  `packages/cli/src/registries/capabilityContract.ts`) is the
  authoritative location, or whether the schema should move to
  a surface-neutral package that mobile and web can import
  directly without depending on the CLI. The CLI-coupled home
  may not be the right dependency direction for a load-bearing
  contract.
- How surfaces enumerate canonical triggers at runtime. Schema
  introspection works for CLI; mobile needs a manifest fetch
  with stale-while-revalidate; web needs a build-time generation
  step. Discovery is not the same problem on every surface.
- Cross-surface handoff UX — explicit "continue on CLI" deep
  link from mobile vs. implicit "any open surface with the
  project loaded can pick it up." Implicit matches the
  design's spirit but is harder to debug when something goes
  wrong.
- What happens when CLI initiates a long-running realisera and
  the user closes their terminal. The design implies the run
  continues; it does not yet say where, and that is a real gap
  in the "single owner per run" model.

## Confidence

provisional — D68's "one workflow" is honored in spirit, but the
schema-ownership, conflict-resolution, and in-flight run-ownership
questions above could each reshape the design. Promote to firm
once the conflict-resolution policy and schema-location questions
are decided; the rest of the open list can be tracked alongside
implementation.
