# Plan: CI Gating for ISS-31

<!-- Level: light | Created: 2026-04-11 | Status: active -->

## What

Add a GitHub Actions workflow that runs the existing test suite (`pytest`) and the spec linter (`python3 scripts/validate_spec.py`) on every push to main and every pull request targeting main. No matrix testing, no coverage reporting, no new dependencies.

## Why

The test suite has 240 tests covering all 18 linter checks, all 8 skill scripts, hooks, the eval runner, and the contract generation script. Without CI enforcement, these tests are advisory: a change can silently break a linter check or a skill script and reach main undetected. ISS-31 was deferred four times because the suite was not mature enough; it now is. CI closes the gap between "tests exist" and "tests are enforced."

## Constraints

- No external dependencies to install (Python stdlib only; no pip step needed)
- No secrets or credentials required
- `docs/chore/test = no bump` per semver policy — no version bump
- Single workflow file only; do not add matrix builds, coverage upload, or badge markup

## Acceptance Criteria

▸ GIVEN a push or pull request targeting main WHEN the CI workflow runs THEN pytest passes with all tests green AND validate_spec.py exits with 0 errors, 0 warnings
▸ GIVEN a change that breaks one or more tests WHEN a pull request is opened THEN the CI job fails and the PR shows a blocking check status
▸ GIVEN a change that introduces a spec linter error WHEN a pull request is opened THEN the CI job fails at the validate_spec.py step
▸ GIVEN the workflow file is committed to main WHEN viewed in the GitHub Actions tab THEN the workflow is listed and the most recent run shows a passing status
