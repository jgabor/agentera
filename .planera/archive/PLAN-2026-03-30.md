## What

Evolve DOCS.md from a flat documentation index into a three-layer convention contract
(conventions, artifact mapping, index) that all skills consult before reading or writing
artifacts. Update dokumentera to perform a first-run survey that proposes conventions from
codebase observation. Update all consuming skills to check DOCS.md for artifact paths with
root as fallback.

## Why

Skills currently impose their own structure on host projects — dropping up to 10 markdown
files in the project root. When deploying into codebases with existing doc conventions
(docs/, shared/docs/, auto-generated API docs), this breaks the project's organizational
contract. Per the decision profile's "convention over configuration" principle (high confidence),
the project's convention must win.

## Constraints

- Backward compatible: root placement must still work when no DOCS.md exists
- All skills work standalone — no hard dependency on DOCS.md
- DOCS.md is the only coordination mechanism (no separate config files)
- Dokumentera owns DOCS.md creation and maintenance; other skills only read the artifact
  mapping section
- Artifact resolution pattern must be compact — no more than 5 lines of instruction per
  skill, to minimize token budget impact on already-long SKILL.md files
- When DOCS.md is created after artifacts already exist at root, dokumentera must detect
  the existing artifacts and offer to relocate them to the newly mapped paths
- Directory-based skill state (.optimera/harness, .planera/archive/, .visionera/) is
  out of scope — these are operational skill-internal state, not project documentation

## Scope

**In**: DOCS.md template redesign, dokumentera workflow update, artifact path resolution
for 8 consuming skills (visionera, resonera, realisera, optimera, planera, inspektera,
inspirera, dokumentera)

**Out**: profilera (writes to ~/.claude/profile/, not project root), directory-based skill
state (.optimera/, .planera/, .visionera/), migration CLI tooling

**Deferred**: automated migration tooling that bulk-relocates artifacts across many projects

## Design

Three-layer DOCS.md structure: a conventions section declaring where docs live and how
they're styled, an artifact mapping section providing a lookup table from artifact name
to file path, and an index section tracking all documentation (authored, generated,
skill-produced). Skills consult the artifact mapping before every read or write, falling
back to root when DOCS.md is absent or silent on a given artifact. Dokumentera populates
DOCS.md via a first-run survey that observes the codebase and proposes conventions.

## Tasks

### Task 1: Design the three-layer DOCS.md template

**Depends on**: none
**Status**: complete
**Acceptance**:
- GIVEN the DOCS.md template WHEN a reader examines it THEN they find three distinct
  sections: conventions, artifact mapping, and documentation index
- GIVEN the conventions section WHEN populated THEN it declares a doc root path, style
  defaults (tone, structure), and any auto-generated doc tooling present in the project
- GIVEN the artifact mapping section WHEN a skill looks up an artifact by its canonical
  filename (e.g., VISION.md, PLAN.md) THEN it finds either an explicit path or no entry
  (triggering root fallback) — no ambiguity in lookup key
- GIVEN the index section WHEN dokumentera audits docs THEN it can track documents with
  status values: current, stale, missing, intent, and generated

### Task 2: Update dokumentera with first-run survey and convention support

**Depends on**: Task 1
**Status**: complete
**Acceptance**:
- GIVEN a project with no DOCS.md WHEN dokumentera runs THEN the user is presented with
  a proposed three-layer convention map based on observed codebase structure, and can
  approve or modify it before it is written
- GIVEN a project with docs in a non-root location WHEN the survey completes THEN the
  proposed artifact mapping is consistent with the project's existing doc organization
- GIVEN a project with auto-generated docs WHEN the survey completes THEN those docs
  appear in the index with 'generated' status
- GIVEN skill artifacts already at root WHEN DOCS.md is created with non-root artifact
  paths THEN dokumentera detects the existing root artifacts and offers to relocate them
- GIVEN DOCS.md style defaults WHEN dokumentera writes new documentation THEN the output
  matches declared tone and structure, with fine details inferred from existing docs

### Task 3: Define canonical artifact resolution pattern

**Depends on**: Task 1
**Status**: complete
**Acceptance**:
- GIVEN a draft resolution pattern WHEN added to one representative skill THEN it covers
  all artifact reads and writes for that skill (both its own artifacts and cross-skill
  reads like reading HEALTH.md maintained by inspektera)
- GIVEN the resolution pattern WHEN measured THEN it is no more than 5 lines of
  instruction text
- GIVEN a project with DOCS.md artifact mapping WHEN the skill runs THEN every artifact
  access uses the mapped path
- GIVEN a project without DOCS.md WHEN the skill runs THEN every artifact access falls
  back to root with no errors or warnings

### Task 4: Propagate artifact resolution to all remaining consuming skills

**Depends on**: Task 3
**Status**: complete
**Acceptance**:
- GIVEN the canonical pattern from Task 3 WHEN applied to all 8 consuming skills THEN
  the wording and logic are identical across skills
- GIVEN any consuming skill (visionera, resonera, realisera, optimera, planera,
  inspektera, inspirera, dokumentera) WHEN it reads or writes any artifact in a project
  with DOCS.md THEN it uses the mapped path
- GIVEN any consuming skill WHEN it reads or writes any artifact in a project without
  DOCS.md THEN it falls back to root (backward compatible)

### Task 5: Update repo documentation and create reference DOCS.md

**Depends on**: Tasks 1, 2, 3, 4
**Status**: complete
**Acceptance**:
- GIVEN the README state artifacts table WHEN a reader looks at the DOCS.md row THEN
  they understand its three-layer role (conventions, artifact mapping, index)
- GIVEN the CLAUDE.md ecosystem section WHEN a contributor reads it THEN they understand
  that skills check DOCS.md for artifact paths with root as fallback
- GIVEN this repo's own DOCS.md WHEN read as a reference implementation THEN it
  demonstrates all three layers with conventions appropriate to this repo

## Overall Acceptance

- GIVEN a project with existing doc conventions WHEN the full skill suite is deployed
  THEN skill artifacts are placed according to the project's conventions, not hardcoded
  to root
- GIVEN a new project with no existing conventions WHEN the skill suite is deployed
  THEN artifacts default to root placement (current behavior preserved)
- GIVEN any consuming skill in the suite WHEN it reads or writes any artifact (its own
  or cross-skill) THEN it checks DOCS.md first and falls back to root — no skill
  bypasses the convention map
